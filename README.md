# Домашнее задание к занятию "Базы данных, их типы" - Валик Александр



### Задание 1

1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.

Тип СУБД: Реляционная (SQL) – например, PostgreSQL, Oracle, MS SQL Server, требуется целостность данных (ACID) и чёткая схема.

2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам.

Тип СУБД:

Для лендингов – документоориентированная NoSQL (например, MongoDB) или ключ-значение (Redis для временных кешей, Aerospike).
Лендинги могут часто менять структуру форм, полей; важна гибкость и высокая скорость записи.

Для CRM – реляционная (PostgreSQL) для консистентной работы с клиентами, сделками, задачами. Либо гибридная (например, PostgreSQL с JSONB-полями для частично изменчивых атрибутов).

3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании.

Тип СУБД: Иерархическая или документоориентированная с поддержкой деревьев – например, MongoDB (модель “один документ – одна страница”), или даже файловое хранилище + мета-база (например, SQLite для простоты). Данные чётко структурированы по иерархии компании (отдел → подраздел → документ).

4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов.

Тип СУБД: Графовая СУБД – например, Neo4j, ArangoDB (мультимодельная с графами). Задача поиска кратчайших маршрутов, распределения по узлам (точки доставки, склады) – это классика графов (вершины = точки, рёбра = расстояния/время).

### Задание 2

Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно.

Действия, предпринимаемые пользователем:

1. Авторизация в мобильном приложении (на сайте) своего банка или поиск банкомата и ввод пин-кода.
2. Проверка наличия необходимой суммы на счете карты.
3. Выбор мобильного оператора для пополнения баланса.
4. Ввод номера мобильного телефона.
5. Ввод суммы, необходимой для пополнения баланса.
6. Проверка поступления средств на баланс счета телефона.

Действия с точки зрения платежных и ИТ-систем:

1.	Инициация и валидация входных данных (номера телефона и суммы).
2.	Блокировка средств на источнике платежа (банк/кошелёк).
3.	Передача запроса оператору связи, зачисление на баланс.
4.	Фиксация успеха оператором и подтверждение платёжной системе.
5.	Списание средств с пользователя и зачсление на баланс.
6.	Уведомление пользователя и завершение транзакции.

### Задание 4


1. В кластере из 1000 узлов сеть — самый узкий ресурс. Если СУБД часто перемещает данные между узлами (shuffle), производительность упадёт катастрофически.

Следовательно, выбирается shared-nothing СУБД, где:

 * каждый узел хранит только свою часть данных (партицию / шард)

 * вычисление переносится к данным, а не наоборот

 * нет единого дискового хранилища (SAN) и нет единого master-узла для всех запросов

Greenplum (PostgreSQL-совместимая, shared-nothing, до 1000+ узлов)

2. Модель: DAG (Directed Acyclic Graph) + распределённые партиции + локальное выполнение

Почему она лучшая для 1000 машин:
| Характеристика | Почему важна для 1000 узлов |
| ----------------------------------------------- | -------------------------------------- |
| Разбиение задачи на независимые стадии | можно параллелить на все узлы |
| Сохранение промежуточных результатов | при отказе узла не нужно пересчитывать всё |
| Планировщик, учитывающий расположение данных | минимизирует пересылку данных по сети |
| Fault tolerance через lineage (линию происхождения) | вместо дорогих контрольных точек |
